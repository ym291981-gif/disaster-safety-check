# Google Apps Script（集計 API）

スプレッドシートに紐づけた GAS で、**指定セッションの集計結果を JSON で返す（GET）**ほか、**回答をレスポンスシートに保存する（POST）**ができます。

## 手順

1. **スプレッドシートを開く**（社員マスター・セッション・レスポンスの3タブがあるブック）。
2. **拡張機能 → Apps Script** でエディタを開く。
3. デフォルトの `Code.gs` の中身をすべて削除し、このフォルダの **`Code.gs` の内容をコピーして貼り付け**る。
4. **保存**（ディスクアイコン）し、左上の「実行」で `doGet` を一度選んで実行すると、初回だけ **権限承認**（スプレッドシートの参照）を求められる。承認する。
5. **デプロイ → 新しいデプロイ** → 種類「ウェブアプリ」  
   - 次のユーザーとして実行: **自分**  
   - アクセスできるユーザー: **全員**（課題・検証用。本番は要検討）  
   - デプロイして **ウェブアプリの URL** を控える。
6. ブラウザで次を開く（`202605` など、セッションシートの ID 値に置き換え）:

   `https://script.google.com/macros/s/.../exec?sessionId=202605`  
   クエリ名は **`sessionId`（推奨）** 。`sessionID` や `session_id` でも可（スクリプトが吸収）。

   セッション列名が **`session_ID`** の場合も対応済み（`Code.gs` の `sessionIdFromRow_`）。

7. **JSON が表示されれば成功**。

**コードを更新したあと**は、必ず **デプロイ → デプロイを管理 → 右の鉛筆（編集）→ バージョンを「新バージョン」→ デプロイ** でウェブアプリを更新する（更新しないと古い `doGet` / `doPost` のまま動く）。

## 回答を送るまでの手順（初心者向け）

ここでは「**ブラウザのフォームから送信 → スプレッドシートのレスポンスに1行書く**」までを説明します。難しい `fetch` や CORS は使わず、**普通のフォーム送信（POST）**だけです。

### 全体の流れ（イメージ）

1. あなたが **回答用の HTML**（`response-ui/respond.html`）をブラウザで開く。  
2. 画面上で **ウェブアプリの URL・セッション ID・社員番号・ステータス** を選んで「送信」する。  
3. ブラウザが **Google の GAS の URL に POST** する。  
4. GAS の **`doPost`** が動き、**同じブックの「レスポンス」シートを開いて行を追加（または上書き）**する。  
5. 送信後、画面には **JSON**（`"ok":true` など）がそのまま表示される。スプレッドシートを開くと **行が増えている／値が変わっている**。

### 手順 1: GAS に最新の `Code.gs` を入れる

1. スプレッドシートから **拡張機能 → Apps Script** を開く。  
2. エディタの **`Code.gs`** を、リポジトリの **`google-apps-script/Code.gs` と同じ内容**にする（すでに入っているなら、**`doPost` や `saveResponse_` が含まれているか**だけ確認）。  
3. **保存**する。

### 手順 2: ウェブアプリを「再デプロイ」する

POST も **GET と同じウェブアプリ URL**（`.../exec`）で受け付けます。コードを変えたら **必ず再デプロイ**してください。

1. 右上 **デプロイ → デプロイを管理**。  
2. いま使っているウェブアプリの **右端の鉛筆（編集）**。  
3. **バージョン**を **「新バージョン」** にする。  
4. **デプロイ** を押す。

### 手順 3: `respond.html` を開く

- **GitHub Pages** を使っている場合の例:  
  `https://（ユーザー名）.github.io/（リポジトリ名）/response-ui/respond.html`  
- ローカルで `serve` している場合の例:  
  `http://localhost:3000/response-ui/respond.html`

**本番で「ウェブアプリ URL」を毎回入力させない:** `response-ui/respond.html` 内の **`DEFAULT_GAS_WEBAPP_URL`** に、手順 2 の **`.../exec`** を文字列で書いて保存し、GitHub Pages 等にデプロイする。設定済みのときは画面上の URL 欄は出ません。デプロイ URL が変わったらこの定数を更新する。緊急時のみ、ページに **`?gasUrl=`** または **`?exec=`** で上書き可能。

### 手順 4: フォームに入力して送信する

1. **ウェブアプリ URL** … **`DEFAULT_GAS_WEBAPP_URL` が空のときだけ**表示されます。手順 2 で控えた **`https://script.google.com/macros/s/.../exec`**（`?` 以降は付けない）を貼る。埋め込み済みならこの手順は不要。  
2. **セッション ID** … トークン無しの手動モードのときのみ。**トークン付き URL では表示されません。**  
3. **社員番号** … 手動モードのときのみ。トークン付きでは不要。  
4. **ステータス** … ラジオで1つ選ぶ（内部では `safe` など英字で送られます）。  
5. **その他**を選んだときは **コメント必須**（GAS がチェックします）。  
6. **「この内容で送信する」** を押す。

**補足:** アドレス欄にクエリを付けると自動入力されます。

`?sessionId=202605&employeeId=1001`

（`employeeId` は `employee_id` でも可。）

### 手順 5: 成功かどうかを見る

- 送信後のページに **`"ok":true`** と出ていれば成功です。  
- **`"ok":false`** と **`error`** が出ていれば、メッセージを読んで（セッション不一致・社員番号なし・コメント不足など）直してください。

### 手順 6: スプレッドシートで確認する

1. **レスポンス** タブを開く。  
2. 該当する **session_id + 社員番号** の行が **新しく追加されている**か、**既存行のステータス・日時が更新されている**かを確認する。  
3. **集計ダッシュボード**（`dashboard.html`）で同じセッション ID を読み込み直し、人数が変わるか確かめる。

### よくあるつまずき

| 症状 | 確認すること |
|------|----------------|
| 送信後ずっと真っ白 | URL が `/exec` まで含まれているか。ポップアップブロックはないか。 |
| `session not found` | セッション ID が **セッションシートの ID と完全一致**か。 |
| 社員マスターに存在しません | 社員番号の打ち間違い、またはマスタに行がない。 |
| その他はコメント必須 | 「その他」選択時にコメントを書いたか。 |

本番では **URL に社員番号を載せない** ようにし、**トークン1本でサーバ側が本人を特定する**のが望ましいです。以下で、そのやり方を説明します（検証用の **手動モード** も引き続き使えます）。

## 回答トークン（本人確認つき URL）

### 用語の整理（読みながら進められます）

| 用語 | 説明 |
|------|------|
| **トークン（token）** | URL に載せる **長いランダムな文字列**。第三者が推測しにくく、「このリンクを開いた人＝この社員・このセッション」とサーバが結び付ける鍵です。パスワードではありませんが、知られればなりすましにつながるので **メールやチャットで個別送付**するのが基本です。 |
| **回答トークンシート** | スプレッドシートのタブの1つ。**トークン・セッション ID・社員番号** の対応表を保存します。GAS が POST されたトークンをここで照合します。 |
| **`issueTokensForSession`** | Apps Script エディタから **手動実行する関数**。指定したセッションについて、社員マスターの**在籍者ひとりにつき1行**トークンを発行し、シートに書き込みます（同じセッションの古いトークン行は削除してから再発行します）。 |
| **`tokenInfo`** | 回答ページが「どのセッションか」を表示するための軽い API。**GET** で `?token=...&mode=tokenInfo` と呼びます。**社員番号は返しません**（セッション ID とタイトルなどのみ）。 |
| **`TOKEN_MARK_USED_AFTER_SAVE`** | `Code.gs` 上部の設定。**`true`** にすると、トークンで一度回答が保存されると「使用済み」になり **同じリンクでは再送信できなくなります**（ワンタイム用途）。**`false`**（既定）は検証しやすく、何度でも送信できます。 |
| **ウェブアプリ URL** | GAS を「ウェブアプリ」として公開した **`.../exec`** までの URL。フォーム POST と JSON GET の両方がこのエンドポイントに届きます。 |

### 事前準備（一度だけ）

1. スプレッドシートに **`回答トークン`** という名前のシートを追加する。
2. **`回答トークン` シートの 1 行目だけ**に、列の「名前札」（**ヘッダー**）を書く。**2 行目以降は空のままでよい**です（後からスクリプトが自動で埋めます）。

   - **どこに書くか**: スプレッドシートで一番上の行（行番号 **1**）。例として **A1 に `token`**、**B1 に `session_id`**、**C1 に `社員番号`** と並べてもよいし、列の順番は入れ替えても構いません。
   - **なぜ必要か**: GAS は「どの列がトークンか」を **1 行目の文字列**で探します。ここが空や別名だと、発行も照合も動きません。
   - **最低限そろえる名前**（どれか一方の表記で OK）:

   | 推奨ヘッダー | 役割 |
   |--------------|------|
   | `token`（または `トークン`） | 発行した文字列 |
   | `session_id`（または `セッションID` / `sessionId`） | セッションシートと同じ ID |
   | `社員番号`（または `employee_id`） | 対象者 |
   | `used`（または `使用済み`） | 任意。**`TOKEN_MARK_USED_AFTER_SAVE = true`** のときに使用済みかどうかを書く |
   | `expires_at`（または `有効期限`） | 任意。**日時**を入れると、その時刻を過ぎたトークンは拒否される |

3. **`Code.gs` を最新にして保存**し、**ウェブアプリを再デプロイ**する（「新バージョン」）。

### セッションごとの流れ（運用）

1. **セッションシート**に対象の安否確認セッションが登録されていることを確認する（例: ID が `202605`）。
2. **トークンを一括発行する（`issueTokensForSession` の実行）**  
   ここでやることは一言でいうと、「**このセッションについて、在籍社員全員ぶんの秘密のリンク用文字列をシートに書き出す**」です。次の順で進めると迷いにくいです。

   **2-a. スプレッドシートからスクリプトを開く**  
   対象のブックを開いた状態で、メニュー **拡張機能 → Apps Script**。別タブにコード編集画面が開きます。

   **2-b. いまのセッション ID を決める**  
   **`セッション`** シートに書いてある **セッション ID** と **一字一句同じ**文字列をメモする（例: `202605`。`2026_5` とは別物です）。

   **2-c. なぜ「実行」がややこしいか（ここだけ読めば OK）**  
   **`issueTokensForSession`** は内部的には「**セッション ID を 1 つ決めてから、そのセッション用のトークンを全員分発行する**」関数です。ところが Apps Script の画面上部にある **▶ 実行**は、基本 **「引数の入力欄」がありません**。そのため **`issueTokensForSession` をプルダウンから選んで実行しても、セッション ID が渡らずエラー**になりやすいです。**対処**: 引数なしで実行できる **`runIssueTokensFromEditor`** を使います（`Code.gs` に同梱）。

   **2-d. コードでセッション ID を書き込む**  
   `Code.gs` の **`SESSION_ID_FOR_TOKEN_ISSUE`** の **`'202605'`** を、手順 **2-b** でメモした ID に **書き換える**（**セッションシートと同じ文字列・半角引用符で囲む**）。

   **2-e. 保存して実行する**  
   **保存**（ディスクアイコンまたは Ctrl+S）。エディタ **上部の関数プルダウン**（ツールバー付近のドロップダウン）で **`runIssueTokensFromEditor`** を選び、**▶ 実行**。  
   初回のみ **権限の確認**が出たら、このブックへのアクセスを許可する。

   **2-f. うまくいったか確認する**  
   - **`回答トークン`** シートで、**2 行目以降**に **在籍人数ぶんの行**が増え、`token`・`session_id`・社員 ID 列が埋まっているか確認します（同じセッションで **再実行**すると、そのセッションの **古いトークン行がいったん消えてから**、全員分が **入れ直されます**）。  
   - エディタでは **エラーなく実行完了**なら成功です。トークンの文字列はシートの `token` 列を見てください。

   **（別手段）** 自分で関数を足したい場合は、`issueTokensForSession('あなたのセッションID');` を呼ぶだけの関数を `Code.gs` に書いて、その関数を ▶ 実行しても同じです。

3. **各人に送る URL** を組み立てる（静的サイトの例）。

   `https://（あなたの Pages のベース）/response-ui/respond.html?token=（ログまたはシートの token 列の値）`

   **ウェブアプリ URL** は回答フォーム画面でユーザーが一度入力すればブラウザに保存されます（または説明文で会社共通の `.../exec` を案内）。

4. 受信者がリンクを開くと **トークンモード**になり、**セッション ID・社員番号の入力欄は出ません**。ステータスとコメントだけ選んで送信します。
5. 送信後、ページに **`"ok":true`** が表示され、**レスポンス**シートにその社員の行が追加または更新されます。サーバは **トークンから session_id と社員番号を復元**するので、フォームに載せる必要はありません。
6. （オプション）本番で **1回きりのリンク**にしたいときは `TOKEN_MARK_USED_AFTER_SAVE` を **`true`** に変更して再デプロイし、再度 `issueTokensForSession` でトークンを発行する。

### トラブルシュート（トークン）

| 症状 | 確認すること |
|------|----------------|
| トークンが無効 | `回答トークン` シートにその文字列があるか。セッション ID・社員番号列が空でないか。 |
| 使用済みエラー | `TOKEN_MARK_USED_AFTER_SAVE` が true で既に送信済みか。必要なら再発行。 |
| 期限切れ | `expires_at` が現在より過去になっていないか。 |
| tokenInfo が失敗 | GET の URL が **`.../exec`** ベースか。トークンにスペースや切り捨てがないか。 |

## 次の段階: 集計ダッシュボード（HTML）

リポジトリの **`admin-ui/dashboard.html`** をブラウザで開き、次を行う。

1. プロジェクトのルートでターミナルを開き、例: **`npx --yes serve .`** または **`npx --yes serve admin-ui`** でローカル HTTP サーバーを起動する（`file://` 直開きだと `fetch` が CORS で失敗することがある）。
2. ブラウザで `http://localhost:3000/admin-ui/dashboard.html`（ポートは `serve` の表示に合わせる）を開く。
3. **ウェブアプリ URL** に `https://script.google.com/macros/s/.../exec` を貼り、**セッション ID** に `202605` などを入力し、「**集計を読み込む**」を押す。
4. よければ「**URL をブラウザに保存**」で次回の入力を省略。

## シート名・ヘッダー

`Code.gs` 上部の定数は、ブックのタブ名と **1行目のヘッダー文字列**と一致させる。

| 定数 | 想定タブ名 |
|------|------------|
| `SHEET_EMPLOYEES` | 社員マスター |
| `SHEET_SESSIONS` | セッション |
| `SHEET_RESPONSES` | レスポンス |
| `SHEET_TOKENS` | 回答トークン |

**レスポンスの `ステータス` 列**は README の英字（`safe` 等）推奨。`Code.gs` は **日本語（無事・軽症・軽傷・要救助・要救護・その他）** も英字に正規化して集計する。**ステータスが空の行**は「行はあるが未回答」として扱う。

**セッション ID** は、**セッションシート・レスポンスシート・ダッシュボード入力の3か所で完全一致**させる（`202605` と `2026_5` は別物）。ブラウザの GAS URL は `?sessionId=2026_5` のように **レスポンスの `session_id` と同じ文字列**を指定する。

**在籍フラグ**（社員マスター）が `はい` / `1` / `TRUE` / `在籍` のいずれかなら対象者に含める。列がない・空なら全員を対象とする。

## 返却 JSON の形（概要）

`session`, `totals`（対象・回答済・未回答・要救護）, `byStatus`, `needHelpList`, `noResponseList`, `byDepartment`（部署別の回答済／未回答件数）を含む。詳細は `Code.gs` 内の `buildPayload_` を参照。
