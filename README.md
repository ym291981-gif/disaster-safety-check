# 災害時安否確認システム（HTML図解型）

災害発生時に社員へ即座に安否確認通知を送り、回答結果を自動集計し、管理者へ **HTML図解ダッシュボード** として状況共有するシステム。

## コンセプト

「通知する仕組み」ではなく、**災害時に誰を優先して助けるべきか即判断できる仕組み**。

## 最優先ゴール

- 要救護者の即時把握
- 未回答者の把握
- 管理者による初動対応の迅速化
- 情報を視覚的に一目で理解できる状態にする
- 回答者が最小操作で回答できること
- 通知対象者の管理を自動化し、人手作業を削減すること

## 背景課題（現行運用）

- メールとLINE WORKSを使った複雑な回答導線
- メールアドレスの手動コピペ管理
- 通知送信準備に時間がかかる
- 回答者 / 未回答者の即時把握が困難
- 要救護者の優先抽出ができない
- 管理者視点の見える化不足
- 災害時に操作数が多すぎる

## LINE運用方針

本来は **LINE WORKS** を想定しているが、会社権限・管理者権限に関わるため、課題制作では **通常のLINE公式アカウント** を利用する。

**理由:** 個人開発しやすい、API連携しやすい、権限制約が少ない、通知体験を再現可能、将来的にLINE WORKSへ移行可能。

---

## 回答チャネル（重要）

社員は **LINE と メールの両方から回答可能** とする。

- LINE利用者は即時回答しやすい
- メール環境しかない人も回答可能
- 通信事情に左右されにくい
- 回答率向上
- 複数導線によるBCP強化

---

## 社員情報・通知対象（重要）

通知対象者の管理は手動コピペではなく、**社員情報データベースから自動取得**する。

**目的:** 手作業削減、通知漏れ防止、最新情報反映、入社/異動/退職への対応、管理コスト削減。

**動作:** 安否確認開始時にDBから通知対象者を取得し、自動でLINE・メール送信を行う。

### 推奨データソース

- **初期実装:** Google Spreadsheet
- **将来拡張:** CSV連携、社内DB連携、人事システムAPI連携

### 社員マスターデータ項目

| 項目 | 内容 |
|------|------|
| employee_id | 社員番号 |
| name | 氏名 |
| department | 部署 |
| office | 拠点 |
| email | メールアドレス |
| phone | 電話番号 |
| line_user_id | LINE識別子 |
| emergency_contact | 緊急連絡先 |
| active | 在籍フラグ |

### レスポンスシートのデータルール

スプレッドシートの「レスポンス」タブ（回答1件を1行で保持する表）について、次をプロジェクトのルールとする。

1. **1セッションにつき1人1行**  
   `セッションID` と `社員番号`（`employee_id` と同一の識別子）の組み合わせは表全体で一意とする。同一の組が再度送られた場合は、実装側で **既存行の上書き更新**（回答日時・更新日時の更新を含む）とする。

2. **`ステータス`の取りうる値**  
   保存値は次の4種類のみとし、スプレッドシート・API・集計ロジックで揺れないようにする（表示ラベルは回答者UIに合わせる）。

| 保存値 | 意味（回答者UIの4択） |
|--------|------------------------|
| `safe` | 無事 |
| `minor_injury` | 軽症（自力で解決可） |
| `need_help` | 要救助（直ちに救援を求めます） |
| `other` | その他（コメント記入） |

**未回答**は、上記4値のいずれでもない別ステータスとして行を増やすのではなく、**対象者に対する回答行の有無**（またはセッションごとの集計）で判定する。

## 基本フロー

1. 災害発生
2. 管理者が安否確認開始
3. DBから対象社員取得
4. LINE・メールへ一斉通知送信
5. 社員がLINEまたはメールから回答ページへアクセス
6. 回答内容送信
7. 自動集計
8. 要救護者即通知
9. 未回答者期限通知
10. 管理画面で全体確認

### トリガー

- **初期実装:** 管理者が「安否確認開始」ボタンを押す
- **将来拡張:** 地震API連携、気象警報API連携、自動起動

## 回答ページ（重要）

LINE通知・メール通知の両方からアクセスできる **スマホ最適化HTMLページ**。

### 目的

災害時でも迷わず回答できる、最短操作で送信できる、高齢者でも使いやすい、スマホで見やすい。

### UI要件

- Tailwind CSS 使用
- スマホファースト
- 大きなボタン
- **3タップ以内完了** を目安
- 見やすい文字サイズ
- シンプル設計
- 誤操作防止

### 表示項目（全体方針）

- **初期画面:** 氏名・部署・拠点は**表示しない**（スクロール・読了負荷を避け、即タップ優先）。本人特定はURLトークン等でサーバ側処理。
- **状況選択:** 下記「回答者側UI（確定仕様）」の4択。
- **その他:** コメント入力と送信前プレビューを表示。

### 回答項目（確定・4択）

| # | 表示文言 | 配色 | 送信フロー |
|---|----------|------|------------|
| 1 | 無事 | 緑 | 選択 → **確認**（「この内容で送信」「選び直す」）→ 送信 |
| 2 | 軽症（自力で解決可） | 黄 | 同上 |
| 3 | 要救助（直ちに救援を求めます） | 赤 | **タップ1回で即送信**（確認画面なし） |
| 4 | その他 | グレー | コメント入力 → **1行プレビュー** → 「送信する」で確定 |

### 送信後処理

- 回答完了メッセージ表示
- 自動集計へ反映
- 管理画面へ即時反映
- 要救護なら即通知

---

## 回答者側UI（確定仕様）

### 誤タップと緊急度のバランス

- **要救助:** 確認を挟まず**最短送信**（遅延・追加タップの負担を避ける）。管理者への即時通知と整合。
- **無事・軽症:** 誤って「無事」と報告するリスクに備え、**「この内容で送信」「選び直す」**の確認画面を表示。
- **その他:** 補足が必要なためコメント入力のあと、**送信前に1行プレビュー**＋**送信**ボタンで確定。

### レイアウト（信号機スタイル）

各選択肢は**左に色付きの丸（信号）**、**右に文言**の1行（1段）構成。縦順は次のとおり（上から無事→軽症→要救助→その他）。**3段目の要救助は親指が届きやすい位置**に配置する想定。

```
(●緑) 無事
(●黄) 軽症（自力で解決可）
(●赤) 要救助（直ちに救援を求めます）
(●灰) その他
```

### Tailwind 図解（静的HTML）

ブラウザで開いてレイアウト・遷移の見取り図として利用する。

- ファイル: [`response-ui/diagram.html`](response-ui/diagram.html)

## 最重要機能

### 1. 要救護者 即時通知

社員が「要救護」を押した瞬間、管理者へ通知。

**例:**

```
【緊急通知】
営業部 山田太郎さんが救護要請しました。
```

### 2. 未回答者通知

回答期限（例: 30分）経過後、自動通知。

**例:**

```
【未回答者 9名】
鈴木 / 高橋 / 田中 ...
```

### 3. HTML図解ダッシュボード

管理者が一目で全体状況を把握できる画面。

#### 上部カード

- 全社員数
- 回答済
- 未回答
- 要救護

#### 色分け

- **緑:** 無事
- **黄:** 軽傷 / 出社不可
- **赤:** 要救護
- **灰:** 未回答

#### グラフ

- 円グラフ（回答割合）
- 棒グラフ（部署別回答率。未回答がどの部署に偏っているかを把握し、初動のフォロー優先や被害エリアの推測の手がかりにする）

#### 一覧

- **要救護者**（最上部表示）例: 🔴 山田 太郎
- **未回答者** 例: ⚪ 鈴木 一郎

#### 詳細表示（要救護者カードをクリック/タップ）

- 氏名、部署、電話番号、緊急連絡先
- 回答時刻、回答チャネル、コメント内容
- 住所（任意）

## 開発が止まったときの再開メモ（実装順の目安）

1. スプレッドシートのマスタ列と「1回の安否確認」のセッションIDを定義する
2. 管理者「開始」→ 対象抽出 → LINE・メールに個別トークン付きURLを送る
3. 回答POST → ストレージ更新
4. 要救護の管理者通知を早期に実装
5. 未回答の期限通知（バッチまたはスケジュール）
6. ダッシュボード（カード → 一覧 → グラフの順）

## GitHub Pages で回答フォームを開くとき

公開設定が **`/`（ルート）** のとき、次のどちらでも回答ページに届きます。

- **`https://〈ユーザー名〉.github.io/〈リポジトリ名〉/`** … ルートの `index.html` から `response-ui/respond.html` へ誘導
- **`https://〈ユーザー名〉.github.io/〈リポジトリ名〉/response-ui/respond.html`** … 回答ページを直接開く

**合っているかの確認**: GitHub リポジトリの **Code** タブで、`response-ui/respond.html` とルートの `index.html` が **同じブランチ**に存在するか見る。Pages の **Branch** がそのブランチと一致しているか、**Folder** が **`/ (root)`** か（`/docs` にしている場合はファイルは `docs/` 以下に置く）。
